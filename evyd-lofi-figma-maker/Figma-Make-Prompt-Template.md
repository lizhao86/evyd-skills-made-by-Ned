# Figma Make Prompt Template

Use this structure for every Figma Make LoFi wireframe prompt generated by the skill.

---

## Prompt Structure

```
Act as a UX Designer. Create a LoFi wireframe for [Platform] — [Screen Name].

**Context**: [1 sentence from User Story Description — what this feature does and why]
**User**: [Role] — [goal they are trying to accomplish on this screen]

**Screen type**: [form / list / dashboard / modal / drawer / empty state / error state / confirmation]
**Layout**: [Mobile 375px / Web 1440px / Tablet 768px] — [scroll: vertical scroll / fixed / paginated]

**Components**:
- [Component Name] — [position: top / center / bottom, full-width / left / right / floating] — [purpose]
- [Component Name] — [position] — [purpose]
- (repeat for all components, ordered top to bottom)

**Interaction context**: [The specific action or decision the user is making on this screen, drawn from AC]

**Style**: LoFi wireframe. Grayscale only. No colors. Use gray placeholder rectangles for images/icons. Placeholder text only (use [Label] or Lorem Ipsum). Sketch-style, low fidelity, no shadows or gradients.
```

## AC Scenario → Screen Mapping

| AC Scenario Type | Screen to Generate |
|------------------|--------------------|
| Happy Path | Primary screen — default / loaded state |
| Alternative Path | Secondary screen or step in a multi-step flow |
| Edge Case | Empty state, zero-data state, boundary condition screen |
| Error Handling | Error state — inline validation, error modal, or failure screen |

## Example

**Input (from User Story)**

- Platform: `App`
- Role: `Admin`
- Goal: `View inherited read-only permissions when assigning permissions to accounts linked with roles`
- Happy Path AC: Admin opens permission assignment screen → sees inherited permissions listed as read-only → assigns additional direct permissions → saves

**Generated Prompt**

```
Act as a UX Designer. Create a LoFi wireframe for App — Permission Assignment Screen.

**Context**: Admins assign permissions to accounts; permissions inherited from linked roles are shown as read-only and cannot be removed.
**User**: Admin — reviewing inherited permissions and assigning additional direct permissions to an account.

**Screen type**: Settings / Permission management form
**Layout**: Mobile 375px — vertical scroll

**Components**:
- Top navigation bar — top, full-width — "Assign Permissions" title + back arrow
- Section header — below nav — "Inherited Permissions (Read-Only)"
- List item card × 3 — stacked, full-width — each: permission name (left) + lock icon + "Inherited" status badge (right)
- Section header — below inherited list — "Additional Permissions"
- List item card × 3 — stacked, full-width — each: permission name (left) + toggle switch off (right)
- Primary CTA button — bottom, full-width — "Save Changes"

**Interaction context**: Admin views locked inherited permissions (cannot toggle), then enables or disables additional direct permissions via toggles before saving.

**Style**: LoFi wireframe. Grayscale only. No colors. Use gray placeholder rectangles for images/icons. Placeholder text only. Sketch-style, low fidelity, no shadows or gradients.
```
